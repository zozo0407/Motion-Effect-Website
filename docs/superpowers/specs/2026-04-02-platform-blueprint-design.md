# AI 驱动特效原型工坊 - 产品与架构蓝图 (Spec)

## 1. 产品愿景与核心定位

本项目旨在打造一个**定向生成、高度自由、且可调节的特效创作平台**。
平台的核心价值在于将大模型（AI）的逻辑生成能力与底层的图形渲染引擎无缝结合，使用户能够通过自然语言对话，快速产出工业级（兼容剪映）的视觉特效资产。

### 1.1 核心业务需求
*   **实时 AI 创作与预览：** 网页端支持 AI 实时编写 Canvas2D、Three.js、WebGL 代码，并在沙盒环境中瞬间渲染预览。
*   **无缝衔接生产管线：** 生成的特效代码能够直接导出，并完美运行在剪映等专业视频编辑软件的底层渲染引擎中。
*   **AI 灵感副驾驶：** AI 不仅是代码生成器，更是创作导师，帮助用户梳理思路、发散灵感、精调参数。

---

## 2. 平台核心架构 (Web 端)

为了实现上述愿景，平台由三个核心模块构成：

### 2.1 AI 导演台 (AI Chat & Prompt Engine)
*   **作用：** 接收用户自然语言指令，结合预设的 System Prompt，生成符合目标引擎规范的 JavaScript 代码。
*   **交互：** 对话式界面，支持多轮迭代与修改。

### 2.2 统一渲染沙盒 (UnifiedRenderer)
*   **作用：** 网页上的“实时放映厅”与“防爆墙”。
*   **核心机制：**
    *   **安全隔离：** 在独立的上下文中执行 AI 生成的未知代码，防止网页崩溃。
    *   **环境模拟 (Mocking)：** 针对剪映等特定环境，`UnifiedRenderer` 负责在沙盒内注入全局变量（如 `tt.getSystemInfoSync`, `tt.createCanvas`），使专有代码能够在普通浏览器中正常绘制。它**绝不限制** AI 的图形学算法和逻辑创意。

### 2.3 参数控制台 (Parameter Wizard)
*   **作用：** 将代码中的核心变量（如颜色、速度、密度）提取为可视化 UI 滑块，供非技术用户手动微调。

---

## 3. MVP 阶段：单文件导出工作流

在第一阶段（MVP）中，为了最快验证“AI 生产 -> 剪映消费”的闭环，我们采取**最小化依赖**的单文件流转策略。

### 3.1 导出链路
1.  用户在 Web 端通过 AI 生成满意的特效（例如基于 Canvas2D 的粒子系统）。
2.  用户点击 **[下载 JS 源码]** 按钮。
3.  系统将当前运行的核心逻辑代码作为 `game.js` 文件提供下载。
4.  用户手动将该 `game.js` 放入预先准备好的剪映特效模板目录（如 `/webGL/minigame/game/`）中进行覆盖测试。

### 3.2 阶段性限制
*   MVP 阶段的特效包为“静态配置”，暂不包含与剪映客户端 UI 面板联动的动态参数。参数调整均在代码层面或 Web 端完成。

---

## 4. 未来演进：适配器架构与 SDK 预留

为了保证平台未来的极强扩展性，架构设计遵循**适配器模式（Adapter Pattern）**。系统的核心数据流必须是高度抽象的。

### 4.1 核心数据契约 (Data Contract)
无论未来对接什么引擎，AI 生成的最终产物必须被标准化为：
`[ 标准化 JS 逻辑代码 ]` + `[ 描述参数的 JSON Schema ]`

### 4.2 演进路线图

*   **Phase 1 (当前 MVP)：** 
    *   渲染器插座：`UnifiedRenderer` (Web 模拟运行)
    *   发货方式：单文件 `game.js` 下载。
*   **Phase 2 (全自动打包)：**
    *   增加 `ZipPackager` 模块。在服务器端根据 JSON Schema 自动生成 `main.scene`、`LumiManager.js` 等外围文件，直接输出完整的 `.zip` 特效包。
*   **Phase 3 (原生 SDK 预览)：**
    *   **架构预留：** 开发 `CapCutSDKAdapter`。
    *   当 Web 平台获得官方 SDK 权限时，直接用 `CapCutSDKAdapter` 替换现有的 `UnifiedRenderer`。系统将标准化数据（JS+JSON）喂给 SDK，实现网页端 100% 像素级对齐的真实环境预览。外层 AI 逻辑与 UI 界面无需任何修改。
